## What this is

**AegisOps AI** — a multi-agent IT incident-response command center built for the UiPath AgentHack 2026 hackathon. A FastAPI backend orchestrates six sequential "CrewAI" agents (intake → log analysis → KB search → root cause → resolution → audit) via a simulated **UiPath Maestro** workflow, with human-in-the-loop manager approval gates. A Next.js dashboard ("SOC" theme) drives it.

## Commands

### Backend (`backend/`)
```bash
pip install -r requirements.txt
python run.py          # creates tables, seeds DB, runs uvicorn on 127.0.0.1:8000 with --reload
```
`run.py` is the only entrypoint for local dev — it calls `seed_database()` then launches `app.main:app`. API docs at `http://localhost:8000/docs`. There is **no test suite** despite `pytest` being in `requirements.txt`.

### Frontend (`frontend/`)
```bash
npm install
npm run dev            # next dev --turbopack, port 3000
npm run build          # next build --turbopack
npm run lint           # eslint
```

### Full stack
```bash
docker compose up --build   # postgres + redis + chroma + backend + frontend
```

## Critical: Simulation Mode

`SIMULATION_MODE` (env, **default `true`**) governs the entire agent pipeline. When true (or when `OPENAI_API_KEY` is empty), **no real LLM/CrewAI/ChromaDB calls happen** — everything is generated by deterministic simulation code. This is intentional so the demo runs with zero external dependencies or API keys.

- Real CrewAI execution (`_run_crewai_actual`) only runs when `SIMULATION_MODE=false` **and** `OPENAI_API_KEY` is set; on any exception it silently falls back to simulation.
- Real ChromaDB (`RAGService`) only initializes when `SIMULATION_MODE=false`; otherwise the keyword-matching SQL fallback in `query_knowledge_base` is used.

### Demo behavior is driven by incident-title keywords
`backend/app/services/agents.py::_run_simulation_flow` branches on substrings in the incident **title** — this is how demos trigger specific outcomes:
- `auth`/`login` → critical severity, 94% confidence (triggers approval gate via critical severity)
- `billing`/`stripe` → critical, 72% confidence
- `disk`/`space`/`storage` → medium, 99% confidence (auto-remediates, no approval)
- `unsure`/`mystery`/`strange` → forces confidence to 62% (triggers approval gate via <70% rule)
- `retry` in title → forces one agent failure to demo the retry loop in `_execute_agent_steps_with_retry`

## Architecture

### The orchestration flow (the heart of the system)
`backend/app/services/uipath_maestro.py::MaestroOrchestrator` is the core. `run_workflow(incident_id, db)` is dispatched as a **FastAPI `BackgroundTask`** when an incident is created (`incidents.py`) and runs synchronously with `time.sleep()` pauses to make agent progress visible in the UI. Flow:

1. Set incident → `investigating`, read uploaded log file, query RAG for SOP context.
2. `_execute_agent_steps_with_retry` runs the 5 analysis agents sequentially, writing an `AgentExecution` + `AgentResult` row per agent, retrying up to 3× on failure.
3. Persist AI outputs (severity, root_cause, confidence, resolution_plan) onto the `Incident`.
4. **Escalation gate**: if `severity == "critical"` OR `confidence_score < 70.0`, set status → `waiting_approval`, create an `Approval` row, and **return (halt)**.
5. Otherwise → `execute_remediation` (runs audit agent, closes incident, refreshes metrics).

The pipeline resumes via `resume_after_approval()`, called from `approvals.py` when a manager approves. Status lifecycle: `open → investigating → waiting_approval → remediating → resolved → closed`.

### Backend layout
- `app/main.py` — FastAPI app, CORS (wide-open for dev), router registration under `/api`.
- `app/core/config.py` — pydantic-settings; `DATABASE_URL` defaults to **SQLite** (`./aegisops.db`), Docker overrides to Postgres.
- `app/db/models.py` — SQLAlchemy models: `User`, `Incident`, `IncidentHistory`, `AgentExecution`, `AgentResult`, `AuditLog`, `Approval`, `Document`, `SystemMetric`, `Notification`.
- `app/api/endpoints/` — `auth`, `incidents`, `approvals`, `audit`, `metrics`, `agents` (chat + status). Auth is JWT (`python-jose`) + bcrypt; `RoleChecker` enforces role-gated routes (e.g. only `manager`/`admin` can approve).
- `app/services/` — `agents.py` (agent definitions + simulation), `rag_service.py` (ChromaDB + SQL fallback + chat responses), `uipath_maestro.py` (orchestrator).

### Frontend layout
- Next.js 15 App Router, React 19, Tailwind v4, TanStack Query, Zustand. Pages under `src/app/`: `login`, dashboard (`page.tsx`), `incidents` (+ `new`, `[id]`), `approvals`, `audit`, `analytics`.
- Auth state in `src/store/authStore.ts` (Zustand, persisted to `localStorage` under `aegisops_auth`).
- `src/components/DashboardLayout.tsx` is the shared shell and hosts the RAG chat widget.

### Seed accounts
`seed_database()` creates 4 users, all with password `password`: `admin`, `manager`, `engineer`, `auditor`. It also seeds SOP `Document` rows used by the RAG fallback. Seeding is idempotent (skips if any user exists), so **to re-seed you must delete `backend/aegisops.db`**.

## Gotchas

- **Frontend hardcodes `http://localhost:8000/api`** in `fetch` calls across `src/app/**`, ignoring the `NEXT_PUBLIC_API_URL` env var set in `docker-compose.yml`. Change all call sites if the backend URL changes.
- `authStore.ts` has a TypeScript error in the `login` signature (`token: str` instead of `token: string`) — pre-existing.
- The `BackgroundTask` receives the **request-scoped `db` Session**, which FastAPI closes after the response returns. Works in practice here because of timing, but is a known antipattern — be careful when modifying the orchestrator's DB lifecycle.
- `chromadb`/`crewai` are heavy native deps; with `SIMULATION_MODE=true` you can run the whole app **without installing them** (the imports are inside the `if not SIMULATION_MODE` branches).
